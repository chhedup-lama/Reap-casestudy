Block 9 — Measurement & Recommendation

Monitoring & Alerting

Phase 1 monitoring focuses on production reliability, workflow continuity, and control failures. Detect issues early, route them to the right owner, and ensure the system fails visibly and safely — not silently.

Phase 1 Principle. Monitor for operational breakage and control failure first. Deeper AI optimisation — model drift, quality scorecards, anomaly detection — comes in later phases.

Phase 1 Monitoring Scope

Four launch-critical layers must be covered from day one:

Infrastructure Health
  • CPU and memory utilisation
  • Service availability
  • Database health
  • Queue depth
Application Health
  • API latency
  • Error rate
  • Failed workflow transitions
  • Processing job failures
Pipeline Health
  • OCR and extraction latency
  • LLM call failures and timeouts
  • Reprocessing failure rate
Control Health
  • Audit log write failures
  • Missing workflow events
  • Notification delivery failures

Alerting Priorities

Phase 1 alerts are limited to signals that directly affect operations or compliance. Any control-plane failure that affects auditability or case progression triggers immediate investigation.

SignalWhy it mattersSeverity
Audit logging failureControl and compliance obligation — cannot proceed without a traceImmediate
Failed workflow state updatesCases may be stuck or invisible to reviewersImmediate
LLM / OCR provider unavailablePipeline halts; fallback mode must activateImmediate

Fallback Monitoring

Monitoring must explicitly detect when the system has entered a degraded operating mode — so the team knows operations are now on a fallback path, not just that something failed:

  • AI service unavailable
  • Extraction failure rate above threshold
  • Queue backlog beyond acceptable limits
  • Manual fallback volume spike

Operational Dashboards

Operations
Queue & Case Health
  • Queue health and case backlog
  • Processing status
  • SLA risk indicators
Engineering
System Health
  • API latency and error rates
  • Dependency health
  • Worker and job failures
Compliance / Product
Control & Exception View
  • Audit event completeness
  • Workflow exceptions
  • Override and fallback trends

Runbooks & Ownership

Each critical alert must have a named owner, severity level, and first-response action. Phase 1 needs clear runbooks for:

  • Service degradation — owner, triage steps, escalation path
  • Queue backlog — threshold definition, ops escalation, SLA communication
  • Provider outage (LLM, OCR) — fallback activation, reviewer communication
  • Audit logging failure — immediate halt decision, compliance notification
  • Workflow blockage requiring manual intervention — ops steps, case recovery

Future Expansion

Later phases add deeper AI observability:

  • Model drift and confidence-distribution monitoring
  • AI quality scorecards by case type
  • Alerting on override-rate or acceptance-rate shifts
  • Cost and token-usage monitoring
  • Deeper tracing across retrieval, reasoning, and reviewer actions
  • Automated anomaly detection and retraining triggers
