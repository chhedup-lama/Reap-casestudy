Block 6 — Governance

Audit & Logging Architecture

Every material workflow action, evidence change, model output, reviewer edit, and final decision is captured in a complete, reconstructable record. The audit layer makes every case explainable months later — regardless of how documents, policies, or models have changed since.

Design Objective

The audit layer must enable replay of any case to answer six questions:

1
What happened?Full event timeline
2
Who or what did it?Human + system attribution
3
What information was available?Evidence snapshot at decision time
4
Which policy version applied?Frozen clause set at decision time
5
What changed during review?Before/after edit history
6
Why was the decision made?Reason codes + override rationale

Three-Layer Architecture

A flat log table is insufficient. The audit design requires three distinct layers.

A

Immutable Event Log

Time-ordered workflow and action events — append-only, tamper-evident

Workflow events Reviewer interactions Model and automation events Control and governance events Hash-chained integrity WORM storage
B

Versioned Domain Objects

Changing business objects stored with full version history

Case record Document record Extraction result Evidence bundle Policy bundle Recommendation object Final decision object
C

Decision Snapshots

Point-in-time bundles frozen at key milestones — used for reconstruction

Review-ready snapshot Pre-escalation snapshot Approval / rejection snapshot Monitoring handoff snapshot

Every event that materially affects case state, evidence, policy interpretation, model behavior, or decision outcome is logged. Only material events enter the durable audit trail — but with enough detail to reconstruct decision context.

Evidence & Provenance Linking

Every material recommendation and final decision links to the evidence available and used at that time. Crucially, the platform links all four evidence states — not just supporting evidence.

Evidence StateWhy it must be linked
Supporting evidenceThe basis for the decision — what the system and reviewer relied on
Missing evidenceDocuments or fields not present at decision time — context for follow-up or risk
Conflicting evidenceInconsistencies across sources — shows the reviewer was aware of them
Overridden evidenceEvidence present but dismissed or deprioritised — prevents selective reconstruction

Evidence Link Fields

FieldPurpose
Source document ID + versionPin evidence to an exact document state
Document typeEnable filtering by evidence category
Extracted field referenceLink to the specific field, not just the document
Snippet / citation locationPage and position for human verification
Evidence timestampWhen this evidence was captured
Confidence score / bandSystem confidence at time of use
Reviewer-corrected statusFlags whether a human overrode the extracted value
Conflict indicatorMarks inconsistency with another source

Why this matters beyond audit. Edit history is the training signal for model improvement, the evidence for second-line compliance review, and the mechanism for distinguishing system output from human judgment. Without structured before/after capture, override data is lost.

Model Output Logging

Model outputs are stored as versioned, attributable artifacts — not transient UI text. The key distinction: the system preserves both the raw model output and the reviewer-used final output, enabling comparison of what the model suggested versus what the human adopted.

Decision Reconstruction & Historical Snapshots

Reconstruction lets an auditor replay the complete decision context months later — without relying on memory or current-state data. Reconstruction uses snapshotted historical context, never recomputed current retrieval or policy content.

🕑

A — Timeline View

Chronological list of all workflow, review, and model events for a case.

  • State transitions with timestamps
  • Every reviewer action
  • Every AI output event
  • Follow-up and re-review cycles
📄

B — Decision Packet View

Compact package summarising everything that went into the final decision.

  • Final disposition + rationale
  • Evidence used
  • Policy clauses used
  • Reviewer actions and overrides
  • Unresolved conflicts, if any
⇆

C — Before / After Comparison

Side-by-side diff view for audit and quality review.

  • System output vs reviewer final
  • Original extraction vs corrected
  • Pre- vs post-follow-up evidence

Policy Version Preservation

Future policy changes must not alter the explanation of a past case. At each material decision point, the platform creates a policy decision snapshot.

What is preserved
  • Policy document ID
  • Policy version
  • Effective dates
  • Clause IDs used
  • Clause text snapshot or hash
  • Policy family / taxonomy
  • Mandatory vs advisory
  • Jurisdiction metadata
Why it matters
  • Historical approvals remain valid under updated policy
  • Rejected cases stay explainable
  • Model outputs can be reproduced faithfully
  • Internal audit can verify the correct rule set was used
Snapshot contains
  • Applicable policy set ID
  • Selected clause IDs
  • Clause summaries shown to reviewer
  • Effective version metadata
  • Retrieval or rules trace
