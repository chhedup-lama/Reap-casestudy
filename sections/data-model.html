Block 4 — Solution Architecture

Data Model & Object Relationships

The platform is anchored on twelve shared objects that create a common contract across every layer — ingestion, extraction, retrieval, review, orchestration, and governance. These objects ensure outputs remain evidence-backed and that every reviewer action, policy reference, and final decision can be fully reconstructed.

Design Principles

The data model is built around five principles that govern every object and relationship.

Case-centric

Every object attaches back to a single Case as the primary workflow unit. No orphaned data, no floating context.

Evidence-linked

Extractions, summaries, recommendations, and decisions all reference their source evidence — nothing floats free.

Version-aware

Policy references, prompts, model versions, and reviewer edits are traceable over time for full audit reconstruction.

Workflow-native

Objects natively support state transitions, open items, tasks, escalations, and multi-step follow-up cycles.

Reusable

Generic enough to power adjacent workflows — KYC, transaction monitoring, merchant onboarding — without redesign.

1

Case

CASE

The top-level object and system anchor. A Case represents one business review instance moving through the KYB workflow. Every document, extracted fact, open item, task, recommendation, and decision attaches to this object. All workflow state resolves here.

FieldTypeDescription
case_iduuidSystem-wide unique identifier and primary workflow anchor
business_namestringLegal entity name of the applicant business
case_typestringDetermines applicable policy partition and checklist set
current_stagestringWhere the case sits in the review lifecycle
statusstringopen / pending_info / escalated / closed
risk_tierstringDrives routing, escalation thresholds, and SLA assignment
jurisdictionstringDetermines which policy partition is retrieved during grounding
source_channelstringHow the case was initiated — portal, email, API, or internal system
assigned_revieweruuidLinks to the active analyst or queue
created_attimestampCase creation time
updated_attimestampLast modification time across any linked object
linked_decision_statusstringDenormalized current decision state for fast status reads
Why it matters — The Case ties together documents, extracted fields, policy retrieval, open items, reviewer actions, and final decisions. All major processing and workflow state must resolve back to this object — there is no meaningful output that exists outside a Case.
2

Document

DOCUMENT

Each file or submitted artifact attached to the case — application forms, identity documents, certificates, bank statements. Documents are the raw source of truth and must remain persistently linked to the Case to support parsing, extraction, retrieval, evidence grounding, and full audit reconstruction.

FieldTypeDescription
document_iduuidUnique document identifier
case_iduuidForeign key to parent Case
source_typestringHow the document arrived — upload, email, API push, registry pull
file_typestringPDF, JPEG, DOCX, etc.
uploaded_attimestampWhen the document was received by the platform
doc_type_classificationstringCertificate of Incorporation, UBO Declaration, Bank Statement, etc.
languagestringDetected document language (ISO 639-1)
statusstringreceived → parsed → failed / superseded
storage_refstringPointer to encrypted blob storage location
parsed_text_refstringReference to the extracted text layer used for downstream processing
Why it matters — Documents are the raw source of truth. Every extracted field, evidence snippet, and checklist finding must trace back to a specific document. The persistent link to the Case is what makes audit reconstruction possible — even after a review is completed or a decision is appealed.
3

Extracted Field

EXTRACTED_FIELD

Structured values derived from documents by the extraction pipeline. Critically, this object distinguishes between machine-extracted values, normalized values, and reviewer-corrected values — rather than collapsing them into an opaque final value. This distinction is what makes extraction transparent, reviewable, and auditable.

FieldTypeDescription
field_iduuidUnique field identifier
case_iduuidForeign key to parent Case
document_iduuidSource document this field was extracted from
field_namestringSemantic field name — e.g. company_name, registration_number, ubo_ownership_pct
extracted_valuestringRaw machine-extracted value before normalization
confidence_scorefloatExtraction quality signal used in case confidence scoring
normalization_statusstringraw / normalized / failed
validation_statusstringWhether the value has been cross-checked against other documents or registries
evidence_linkuuidLink to the Evidence Snippet that sourced this field
reviewer_editedbooleanTrue when an analyst overrode the machine-extracted value
Why it matters — This object makes extraction transparent and reviewable. It allows the platform to distinguish between what the machine extracted, what was normalized, and what a reviewer corrected — rather than presenting a hidden final value that conflates all three. This distinction is essential for model evals, override analytics, and compliance audits.
4

Entity

ENTITY

A real-world legal or business entity identified within the case — the applicant business, a parent company, a beneficial owner, a director, or a related party. Entity modeling is essential because KYB almost always involves more than one party.

FieldTypeDescription
entity_iduuidUnique entity identifier
entity_typestringapplicant / parent_co / ubo / director / related_party
case_iduuidForeign key to parent Case
namestringLegal name of the entity
registration_idstringCompany registration number or national identifier
addressstringRegistered address
jurisdictionstringCountry or regulatory territory of registration
role_in_casestringHow this entity relates to the review decision
deduplicatedbooleanWhether the entity has been resolved across multiple document references
Why it matters — KYB often involves multiple parties — an applicant, its parent company, beneficial owners, and related parties. Explicit entity modeling supports UBO mapping, cross-document consistency checks, ownership graph construction, and external registry verification. Without it, multi-party cases collapse into unstructured text.
5

Evidence Snippet

EVIDENCE_SNIPPET

The exact source fragment — a passage, clause, table row, or data point — used to support a claim, extraction, checklist finding, summary statement, or recommendation. This is the foundation of explainability. Every major AI output and every reviewer decision links to a specific, retrievable piece of evidence.

FieldTypeDescription
evidence_iduuidUnique evidence identifier
case_iduuidForeign key to parent Case
linked_source_iduuidDocument ID or Policy ID depending on source_type
source_typestringdocument (applicant-submitted) or policy (internal rules)
snippet_textstringThe exact extracted passage or clause text
page_section_refstringPrecise page and section location for human verification
citation_metadatastringAdditional provenance metadata — retrieval method, embedding model version
relevance_scorefloatRetrieval confidence for this snippet against the query context
Why it matters — This is the foundation of explainability. It allows reviewers and auditors to trace every important output — a checklist state, a recommendation, a decision — back to a precise source fragment, rather than trusting opaque model behavior. Without Evidence Snippets, grounded AI assistance becomes unverifiable.
6

Policy

POLICY

The retrieved policy, SOP, or rule artifact used in case assessment. The platform records not just that a policy was applied, but exactly which version and which clause informed a checklist state or recommendation — essential for regulatory defensibility as rules evolve over time.

FieldTypeDescription
policy_iduuidUnique policy identifier
policy_namestringHuman-readable policy or SOP name
section_clausestringGranular rule reference — not just the top-level policy document
jurisdictionstringScopes retrieval to the applicable regulatory context
applicable_case_typestringWhich case types this policy governs
effective_datedateWhen this version became active
versionstringEnables point-in-time reconstruction of past decisions as rules change
source_repository_refstringPointer to the authoritative policy store or document
retrieval_metadatajsonbEmbedding and retrieval provenance for the policy chunk
Why it matters — KYB review is policy-sensitive. The platform needs to know not just that policy was used, but exactly which version and which section informed a recommendation or checklist state. This version-locking is what makes decisions defensible when regulators ask why a specific rule was or was not applied at a particular point in time.
7

Open Item

OPEN_ITEM

A missing document, unresolved discrepancy, unanswered question, or pending clarification. Open Items convert vague review friction into trackable workflow objects with ownership, priority, and due dates — critical for follow-up communication cycles and case continuity across reviewer handoffs.

FieldTypeDescription
open_item_iduuidUnique open item identifier
case_iduuidForeign key to parent Case
typestringmissing_document / discrepancy / clarification_request
severitystringblocking / high / medium / low — determines whether the case can progress
created_bystringsystem (extraction gap detected) or reviewer (manually raised)
statusstringopen / in_progress / resolved / dismissed
linked_evidence_iduuidEvidence or rationale that triggered this open item
owneruuidAssigned reviewer or team queue responsible for resolution
due_datetimestampTarget resolution date for SLA tracking
resolved_attimestampWhen the open item was closed or dismissed
Why it matters — Open Items turn vague review friction into trackable workflow objects. Without them, missing information and discrepancies get lost in unstructured reviewer notes. With them, every gap has an owner, a status, and a resolution path — especially important for follow-up communication, re-review cycles, and handoffs between reviewers.
8

Checklist Item

CHECKLIST_ITEM

A review requirement derived from policy, SOP, or mandatory process controls. Checklisting makes policy application explicit and consistent — creating a structured, auditable bridge between policy retrieval, evidence review, and final decisioning that a reviewer and an auditor can follow step by step.

FieldTypeDescription
checklist_item_iduuidUnique checklist item identifier
case_iduuidForeign key to parent Case
policy_iduuidTraces each requirement back to the exact policy clause that defines it
requirement_namestringHuman-readable requirement description
statusstringpassed / failed / needs_review / blocked
supporting_evidenceuuidEvidence Snippet ID that supports the current status assessment
reviewer_confirmationstringWhether the reviewer confirmed, adjusted, or overrode the AI status
override_reasonstringMandatory annotation when a reviewer bypasses a failed or blocked item
Why it matters — Checklisting makes policy application explicit and consistent across reviewers, teams, and time. It creates a structured bridge between policy retrieval, evidence review, and final decisioning — and the override reason field is what makes departures from policy defensible rather than invisible.
9

Task

TASK

Work assigned to a reviewer, specialist, or queue. Tasks are the operational orchestration layer that carries the review forward when cases involve multiple teams, escalation handoffs, or multi-step follow-up loops.

FieldTypeDescription
task_iduuidUnique task identifier
case_iduuidForeign key to parent Case
task_typestringinitial_review / escalation / re_review / clarification_follow_up
assigned_owneruuidIndividual reviewer or team queue
queuestringstandard / senior / compliance_lead
statusstringpending / in_progress / completed / cancelled
prioritystringhigh / medium / low
sla_deadlinetimestampDrives queue-age alerts and SLA breach tracking
handoff_notestextContext passed when reassigning or escalating the task
Why it matters — Tasks are required for operational orchestration, especially where reviews involve multiple teams, SLA commitments, or escalation follow-up loops. Without a Task object, multi-step reviews collapse into informal communication channels and SLA tracking becomes impossible.
10

Recommendation

RECOMMENDATION

A system-generated suggestion — proceed, request more information, escalate, reject, or monitor. Keeping Recommendation as a separate object from Decision is a deliberate design choice: it makes AI assistance inspectable and measurable, enables acceptance-rate tracking and override analysis, and preserves the clean separation between what the AI suggested and what a human decided.

FieldTypeDescription
recommendation_iduuidUnique recommendation identifier
case_iduuidForeign key to parent Case
recommendation_typestringproceed / request_info / escalate / reject / monitor
rationaletextLLM-generated reasoning, cited to specific Evidence Snippets and Policy clauses
linked_evidenceuuid[]Evidence Snippet IDs that support this recommendation
linked_policy_refsuuid[]Policy IDs and versions that informed this recommendation
confidence_scorefloatModel confidence in this recommendation
reviewer_actionstringaccepted / edited / rejected — powers override and acceptance-rate analytics
Why it matters — This object makes AI assistance inspectable and measurable. It also supports acceptance-rate tracking, override analysis, and safe separation between what the system recommended and what a human decided. This separation is not just good design — it is a regulatory requirement in many jurisdictions where automated recommendations influence compliance outcomes.
11

Decision

DECISION

The final reviewer outcome on the case. There is exactly one current Decision per Case. It is the core business outcome object — and it must be defensible, auditable, and tied to both a specific evidence set and a named reviewer. The override flag records when a reviewer departed from the AI recommendation.

FieldTypeDescription
decision_iduuidUnique decision identifier
case_iduuidForeign key to parent Case (one active Decision per Case)
decision_typestringapprove / reject / pending_info / escalated
reviewer_iduuidIdentity of the deciding reviewer — mandatory for accountability
decided_attimestampWhen the decision was recorded
rationaletextReviewer-authored justification for the outcome
linked_evidence_setuuid[]Evidence Snippet IDs cited in this decision
linked_policy_versionuuidPoint-in-time policy version snapshot applied at decision time
override_flagbooleanTrue when the reviewer departed from the AI recommendation
approval_hierarchystringRecords senior reviewer or compliance lead sign-off chain where applicable
Why it matters — This is the core business outcome object. It must be defensible, auditable, and tied to both a specific evidence set and a named reviewer. The policy version lock ensures that future regulatory review can reconstruct exactly what rules governed the decision — even if the policy has since changed.
12

Audit Event

AUDIT_EVENT

An immutable, append-only record of every system and human action across the case lifecycle. Audit Events make the platform fully reconstructable — Compliance, Ops, and Engineering can answer what happened, when, by whom, and against what version of the model, prompt, and policy.

FieldTypeDescription
audit_event_iduuidUnique audit event identifier
case_iduuidForeign key to parent Case
actor_typestringsystem / user / service — who or what triggered the event
action_typestringWhat occurred — e.g. document_parsed, field_extracted, decision_recorded, reviewer_override
occurred_attimestampPrecise time of the event
changed_object_refstringObject type and ID that was affected by this event
before_after_deltajsonbStructured diff of the state change — before and after values
model_policy_versionstringSnapshot of the AI model and policy version active at the time of the event
Why it matters — Audit Events make the platform reconstructable. They allow Compliance, Ops, and Engineering to understand what happened, when, why, and with what system state. This object is append-only — no Audit Event is ever modified or deleted. It is the platform's memory of itself.

Object Relationships

⚬
Case is the anchor. Every other object resolves back to it. Evidence Snippets are the connective tissue — they link extractions, checklist findings, recommendations, and decisions to their source material. Policy objects are version-locked at decision time so past decisions remain defensible as rules evolve.
FromToCardinalityDescription
Case Document, Extracted Field, Entity 1 → many A case accumulates documents and the fields and entities derived from them
Case Open Item, Checklist Item, Task 1 → many Workflow objects that track progress and gaps through the review lifecycle
Case Recommendation 1 → many Multiple recommendations may be generated across review stages; only the latest is active
Case Decision 1 → 1 One current final decision per case — the core business outcome
Document Extracted Field, Evidence Snippet 1 → many A document yields structured fields and sourced evidence fragments
Evidence Snippet Checklist Item, Recommendation, Decision many → many A snippet may support multiple outputs; an output may cite multiple snippets
Policy Checklist Item, Recommendation, Decision 1 → many Policy clauses define checklist requirements and inform recommendations; version is locked at decision time
Audit Event All objects append-only Generated for every object state change and workflow transition; never modified or deleted
⚙
Why this model matters — This shared data model is what turns the solution from a collection of AI features into a real product platform. It lets every service — ingestion, retrieval, review, orchestration, governance — work against the same system contract. It supports auditability and reviewer control, and makes future extension into adjacent workflows (KYC, transaction monitoring, merchant onboarding) much easier. Without this layer, the product risks becoming a set of disconnected prompts and workflow hacks rather than a scalable operational system.
