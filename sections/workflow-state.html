Block 7 — Solution Architecture

Workflow State Model

The complete KYB case lifecycle — 12 states, all valid transitions, who triggers each, and what gets logged. Designed for a hybrid operating model where automated pipelines handle deterministic steps and humans own every judgment call.

Design Principles

Clear handoff points

Automated and human phases never blur — the boundary between pipeline and analyst is always explicit.

Explicit waiting states

Customer and internal dependencies get their own states, not informal holds inside active review.

Rework without losing trace

Re-review and follow-up loops are first-class states — re-entry is tracked, not hidden.

Escalation path always present

From any active state, escalation to senior review is a valid, structured transition.

Full auditability

Every material state change fires an immutable audit event with actor, trigger, and case version.

State Definitions

All 12 states in the lifecycle, with trigger source, actor type, and the audit event each fires.

State Description Triggered by Type Audit event fires
New Case created; intake processing not yet started API / form submission / onboarding system Automated On case creation
Ingestion Complete Files, metadata, and source inputs received and stored Ingestion pipeline completion Automated On intake completion
Extraction In Progress Document parsing, OCR, entity extraction, classification running Document intelligence pipeline Automated On extraction start
Review-Ready Extraction finished; review packet prepared for analyst System after extraction + validations complete Automated On review package creation
In Review Analyst actively reviewing extracted fields, evidence, and flags Analyst opens / claims case Manual On claim / open / assignment
Follow-Up Requested Reviewer requests more documents, clarification, or corrections Analyst / compliance reviewer Manual On request raised, including reason
Awaiting Response Case paused pending merchant / partner response System after follow-up request sent Automated On outbound request sent
Re-Review New information arrived; case sent back for analyst reassessment System after response received, or analyst reopens Hybrid On re-entry to review cycle
Escalated Case routed to senior compliance / risk / policy team Analyst / rules engine / risk trigger Hybrid On escalation, with reason and target queue
Approved Case approved for onboarding / continuation Analyst / approver / policy reviewer Manual On decision recorded
Rejected Case declined — policy, risk, incomplete response, or adverse findings Analyst / approver / policy reviewer Manual On decision recorded
Monitoring Approved case enters ongoing monitoring / periodic review System after approval Automated On monitoring enrollment
Closed Case complete — no further workflow activity pending System after rejection, monitoring handoff, or admin closure Hybrid On closure

Transition Rules

All valid state transitions grouped by phase. Each transition shows the exact trigger condition and type.

1

Intake & Preparation

All Automated
New → Ingestion Complete
Case payload, uploaded documents, and metadata successfully received and stored Automated
Ingestion Complete → Extraction In Progress
Ingestion service publishes case to extraction / enrichment pipeline Automated
Extraction In Progress → Review-Ready
Extraction, validations, confidence scoring, and packaging complete successfully Automated
Extraction In Progress → Escalated
System detects high-risk flag, sanctions hit, fraud indicator, or unsupported document scenario requiring specialist handling Automated
2

Human Review Cycle

All Manual
Review-Ready → In Review
Analyst claims or opens case from queue Manual
In Review → Follow-Up Requested
Reviewer determines information is insufficient, inconsistent, or missing Manual
In Review → Escalated
Reviewer identifies policy ambiguity, high-risk pattern, fraud concern, or approval authority limit exceeded Manual
In Review → Approved
Reviewer confirms case satisfies KYB requirements and policy thresholds Manual
In Review → Rejected
Reviewer determines case fails policy, risk, or completeness standards Manual
3

Follow-Up & Rework Loop

Mixed
Follow-Up Requested → Awaiting Response
Follow-up request issued to merchant / partner / internal submitter Automated
Awaiting Response → Re-Review
Requested information or documents received before SLA expiry Automated
Awaiting Response → Rejected
No response within SLA, or response materially insufficient after threshold rules applied Manual or Automated
Re-Review → In Review
Analyst resumes case evaluation with updated evidence Manual
Re-Review → Escalated
New evidence introduces heightened risk or unresolved policy conflict Manual or Automated
4

Escalation & Decisioning

All Manual
Escalated → In Review
Specialist reviewer returns case to analyst workflow with disposition notes Manual
Escalated → Approved
Senior reviewer / compliance approver clears case Manual
Escalated → Rejected
Senior reviewer / compliance approver declines case Manual
5

Post-Decision Lifecycle

Mostly Automated
Approved → Monitoring
Approved case handed off to periodic review / transaction monitoring / ongoing KYB controls Automated
Rejected → Closed
Rejection workflow completes — notifications sent, retention steps executed Automated
Monitoring → Closed
Monitoring ownership transferred or case lifecycle formally completed in this workflow Automated or Manual

Audit Event Model

Every material action across the case lifecycle generates an immutable audit event. The audit log is append-only — no event is modified or deleted after creation.

What triggers an audit event

Every state change
Any manual decision or override
Any escalation
Any follow-up request
Any approval or rejection
Any SLA breach or timeout-driven transition
Any reassignment or queue change
Any case reopen / re-review entry

Audit record schema

Each event captures the following fields, locked at write time.

case_id Identifies the case this event belongs to
prior_state State the case was in before this transition
new_state State the case moved into
timestamp Precise time of the event (UTC, immutable)
actor_type system / analyst / compliance / admin
actor_id User ID or service name that triggered the event
trigger_reason_code Structured reason code for the transition
decision_note Free-text comment from the reviewer, if applicable
evidence_snapshot_ref Document version and evidence set active at event time

Manual vs Automated Transitions

Transition typeCoversRationale
Automated Pipeline completion, waiting-state movement, monitoring enrollment, system-triggered risk routing Deterministic processing steps with no judgment requirement — automating these reduces latency and reduces human error
Manual Review start, follow-up initiation, approval, rejection, specialist escalation Judgment-heavy decisions that carry legal and compliance weight — always require named human accountability
Hybrid SLA-triggered rejection, re-review entry, rework loop exits, closure Policy can configure either auto-routing or reviewer-led action based on risk tier, operating model, or product context
⚠
Approval and Rejection are always Manual. No automated transition may move a case directly into Approved or Rejected without an explicit human action and a named reviewer ID — regardless of AI recommendation confidence. This is a hard constraint, not a configuration option.
